# Build Instructions v2 — Final

---

## 1. Product framing

One-line statement: **"A flow-first review workspace that lets a paralegal process hundreds of documents by spending attention only on what's uncertain — not on repeating the same decision hundreds of times."**

Never call it a "PII anonymization tool" in user-facing copy. It's a review workspace.

---

## 2. Tech stack (final)

- **Frontend:** React + Vite + TypeScript, Tailwind CSS.
- **Backend:** Python, FastAPI, SQLite (SQLModel or raw `sqlite3`).
- **Async processing:** FastAPI `BackgroundTasks` + in-memory job state. Frontend polls **one** batch endpoint (`GET /api/status/batch`) every 1.5s, which returns deltas for all in-flight documents in a single payload — not one call per document. This was a real bottleneck in v1; fixed here.
- **PII detection:** Option B (mock fixture) by default. Optional `USE_LIVE_LLM=true` toggle for one document during the demo, never load-bearing.
- **Export:** `zipfile` + `PyMuPDF` for PDF metadata stripping (see section 8).
- **Storage:** files on disk under `/data/uploads/{doc_id}/`; SQLite for everything else.

No auth, no multi-user, no Redis, no websockets.

---

## 3. Document state machine (explicit, single source of truth)

```
UPLOADED → QUEUED → PROCESSING → READY → UNDER_REVIEW → APPROVED → EXPORT_READY → EXPORTED
                ↓ (failure)
           PROCESSING_FAILED → MANUAL_REVIEW_REQUIRED → READY
```

Every document has exactly one state at all times, stored as an enum column, not inferred from other fields. The frontend renders UI purely as a function of this enum — this is what makes Phase 3/4 implementation clean later. Define this enum in a shared `types.ts` / `models.py` pair in Phase 1 before writing any feature code.

---

## 4. Data model (SQLite)

```
documents
  id, filename, content_hash, status (enum above), risk_tier (needs_attention|quick_review|ready),
  risk_score, page_count, ocr_warning (bool), created_at

entities
  id, document_id, text, type, start_offset, end_offset, confidence,
  decision (pending|approved|rejected), cluster_id (nullable), decided_at, decided_by_rule_id (nullable)

clusters
  id, representative_text, entity_type, member_count

rules
  id, description, condition_json, action, created_at

audit_log
  id, entity_id, document_id, timestamp, action, previous_value, new_value, reason (nullable)

session_state
  id (singleton), current_document_id, scroll_position, zoom_level,
  active_filter, active_panel (entities|search|audit), last_updated
```

`content_hash` (SHA-256 of file bytes) is new — it's what powers duplicate detection in Phase 2.

---

## 5. Phase-by-phase build order

### Phase 1 — Foundation (45 min)
Folder structure, the state machine enum, shared types, SQLite schema, API contracts (just the route signatures, no logic yet). Get this skeleton agreed between frontend and backend before writing feature code — this is what the second review's "do it in phases, not feature-by-feature" point is really about: it prevents you from rewriting contracts mid-build.

### Phase 2 — Backend core (60 min)
- Multipart upload endpoint. On each file: compute `content_hash` immediately. If a hash already exists in `documents`, **do not reprocess** — return a `duplicate: true` flag so the frontend can show "Case12.pdf already queued — skip?" instead of silently reprocessing.
- Background task per new document: mock detection → entities → exact-match clustering → risk scoring → state transitions through the machine above.
- `GET /api/status/batch` — single endpoint returning all documents whose status changed since the client's last poll timestamp (pass `?since=` param). This replaces N polling calls with 1.
- `GET /api/search?q=` — search across indexed entity text, return `{entity, document_id, document_filename}[]`. This is built on data you already have (entities table) — no new indexing infra needed, just a `LIKE` query.

### Phase 3 — Frontend skeleton (45 min)
Layout: left nav (Needs Attention / Quick Review / Ready, renamed from High/Medium/Safe — Maya thinks in work, not risk scores), main panel, right sidebar with tabs (Entities / Search / Audit). Persist `active_panel` to `session_state` alongside scroll/zoom/filter — this is the "remembers which sidebar tab was open" detail, basically free since you're already persisting session state.
Build the **empty states now**, not in Phase 7: no documents uploaded ("Drop your case files here"), no results in current bucket, no search results, queue finished. Judges test these on purpose — building them early means they're not a rushed afterthought.

### Phase 4 — Core review loop (2 hrs — biggest time block, protect it)
- Keyboard loop: `J`/`↓` next, `K`/`↑` prev, `A` approve, `R` reject, `Shift+A` approve all remaining in doc, `N`/`P` next/prev document, `Ctrl+Z` undo (20-deep stack), `/` focus search.
- Clustering UI: identical entity text shown as one card with count badge; approve/reject resolves all members.
- **Redaction must mutate state, not just style it.** When an entity is approved (redacted), replace the actual string in the rendered state with `[REDACTED]` — do not leave the original text in the DOM under a CSS black box. This was a real leak in v1; verify by inspecting the rendered HTML, not just the visual.
- Optimistic UI: instant visual update, background sync, rollback only on failure.
- Zero confirmation dialogs anywhere in this phase.
- Macro rules: "always reject text like this" / "always approve this type below X% confidence" — writes to `rules`, applies live to remaining pending entities.
- Bulk-action undo toast: after any action affecting >5 entities, show "47 approved — Undo all" for 8 seconds.

### Phase 5 — Flow features (60 min)
- Focus Resume: scroll/zoom/doc/filter/active_panel restored silently on reload.
- Flow Metrics bar: "18 docs need attention, 139 clear" — not a percentage.
- Global search (built on the Phase 2 endpoint): `/` opens it, typing a name shows "Found in Doc 18, Doc 37, Doc 92," clicking a result jumps straight to that entity in that document.

### Phase 6 — Export (45 min)
- **Partial export is mandatory, not optional.** If 199/200 documents are clean and 1 failed OCR or has pending entities, export the 199 and clearly list the 1 as needing attention — never block the whole batch on one bad file.
- **End-of-Day Safety Check** screen replaces a plain "Export" button: a dashboard showing counts — "184 reviewed, 16 auto-approved, 2 manual-review documents, 1 OCR warning" — with a single "Continue" action. No popup, no "are you sure," just visibility before the irreversible step.
- Metadata scrubbing: before zipping, run exported PDFs through PyMuPDF to strip the metadata dictionary (author, revision history, embedded hidden text layers) — not just draw redaction boxes over visible text. This matters because a redacted-looking PDF with intact metadata is a real legal failure, not a cosmetic one.
- Originals untouched; exports written to a separate `/data/exports/` path; audit CSV included in the ZIP.

### Phase 7 — Polish (45 min)
Loading states, error handling on failed API calls, toast styling consistency, final pass to delete dead code and confirm naming is clean (this is what the "software engineering fundamentals" judging criterion is actually checking).

---

## 6. What's deliberately left out (confirmed correct by both review passes — keep this framing in the writeup)

No real OCR (flag instead), no Redis/Celery, no SSE/websockets (single batch-polling endpoint is sufficient at this scale), no semantic/fuzzy clustering (exact match only), no multi-user/auth, no mobile/responsive layout, no full accessibility audit beyond keyboard-first nav (mention as future work).

---

## 7. Demo script (3.5–4 min)

1. Drag in ~20 files, including one duplicate — show the duplicate-detected prompt (10 sec).
2. Instant entry into workspace, no config screen (5 sec).
3. Needs Attention / Quick Review / Ready buckets — explain Maya reads ~18 docs, not 200 (20 sec).
4. Keyboard loop rapid-fire: J/K/A/R, no mouse (30 sec).
5. Clustered entity: approve once, 14 instances resolve (30 sec).
6. Type a name into search, jump to it across 3 documents (15 sec).
7. Create a macro rule live, watch it apply across the batch (20 sec).
8. `Shift+A`, show the undo toast, Ctrl+Z (20 sec).
9. Refresh mid-session, show silent resume — including which sidebar tab was open (15 sec).
10. Hit the End-of-Day Safety Check screen — point out 1 flagged doc — Continue — partial export still completes for the rest (20 sec).
11. Close on the one-line product statement and your strongest tradeoff call.
